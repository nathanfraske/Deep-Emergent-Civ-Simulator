# Task #43, the derived-output-is-live gate: design-first scope, input-audit, and slice plan

The gate named this arc: a determinism-harness gate that proves a retired-floor derivation's replacement is live, catching a dead (identically-zero or constant) derived output like the `soil_baseline` bug where a 330x sweep of the photosynthesis constants left the state hash byte-identical and nothing noticed. Sibling to the constructor gate (which proves no authored value sits on the canonical path), this proves the derived value that replaced a retired floor is not dead. Design-first: this document grounds the machinery, runs the input-audit, and brings the slice plan. No code until the gate gates the design.

## Grounding: the machinery this extends, and the bug it catches

The constructor gate (`scripts/constructor_gate.py`, `scripts/determinism_gate.py`, PR #153) is a static grep-and-ratchet gate: it proves no new authored `Fixed` constructor or nondeterminism vector appears on the canonical path, against a baseline count. It is a compile-time and text-time gate. The determinism harness (`crates/sim/tests/determinism_harness.rs`) is the runtime counterpart I own: it runs a dawn world, traces the per-tick state hash and event-log hash across a worker-width sweep, and already carries one liveness guard, a floor on the dialogue-move count (`MIN_EXPECTED_MOVES`) so the driven phase can never silently go vacuous. The derived-output-is-live gate is a runtime gate in that harness's family: it perturbs a derivation's inputs and observes whether the derivation responds.

The failure it catches is real and recurring. The `soil_baseline` derivation (`crates/sim/src/environ.rs`) retired the authored floor and derived a replacement, but a prior state left that replacement identically zero, so productivity did not respond to its declared inputs and the state hash was byte-identical under a large sweep, with nothing failing. The same class of bug appeared again in `#141`, where the Dalton evaporation port was identically zero until it was grounded (a four-canonical re-baseline). A derivation that retires a floor and then produces a constant is a silent regression that the constructor gate cannot see, because the constructor gate proves the value is not authored, not that it is alive.

The codebase already marks these derivations with a `@derives:` annotation (seven today, across `celestial.rs`, `locomotion.rs`, `clock.rs`, `environ.rs` water and productivity, `physiology.rs`, and `decompose.rs`). Each names the derived quantity and the inputs it derives from. This is the natural seed of the registry, but a comment is not machine-data, and that gap is the first input-audit finding below.

## The mechanism, in outline

A data-defined registry of retired-floor derivations. Each entry declares the derived quantity, the reserved input or inputs that feed it, and the scenario that exercises it on the run path. The gate, for each entry, perturbs the declared input and asserts the derivation responds. A derivation that does not respond fails, naming itself, the way the dialogue-move floor names a vacuous converse phase.

## The input-audit: two seams, one of them the load-bearing catch

**Seam one, the gate named it: the registry must be data-defined, not a hardcoded list.** A fixed Rust array of derivations authored one level below the derivations themselves would be the same closed-enum defect the value, semantic, institution-function, and provenance substrates each avoid. The registry is a `RetiredFloorDerivation` data row: an id, the derived quantity's handle, the reserved input key or keys to perturb, and the scenario that arms it. The mechanism (perturb, run, assert) is fixed Rust; the membership is data and grows with the world as new derivations retire new floors. The `@derives:` annotations become the human-readable shadow of the machine registry, and a cross-check that every `@derives:` site has a registry row (and the reverse) is itself a cheap static gate, the ratchet the constructor gate already models.

**Seam two, the one I must surface, auditing the gate's own framing: "the run-path state hash MOVES" has a false-positive mode, and I have direct evidence of it.** A derivation can be fully live, its output responding correctly to its input, yet leave the pinned state hash byte-identical, because its effect lands in a downstream deadband or a structurally-zero branch on the pinned scenarios. I hit exactly this twice in the R-UNITS-PIN sigma arc, verified at source: the fine-precision radiant emitter did change its output at every body temperature (about 0.19 percent) yet the four pins held, because the correction sat below the downstream percept discretization; and the metabolic drain radiant term was structurally zero on every pinned path (setpoint at or below ambient, the zero-emission branch), so no perturbation of sigma moved any pin, and yet the derivation was alive in the regime the pinned scenarios do not reach. Under the gate as stated, both would be flagged dead. They are not.

The `soil_baseline` bug was truly dead in a stronger sense: the derived output value was constant (identically zero) regardless of its inputs, so no perturbation moved the value at its own site, not the world hash and not the value itself. That is the distinction the gate must key on:

- Dead, and must fail: the derived output value is constant regardless of its declared inputs. Perturbing the input does not change the derived value at its site.
- Live but byte-neutral, and must not fail: the derived output value does change under input perturbation, but the change does not reach the pinned state hash. This is a coverage gap in the pinned scenarios, not a dead derivation.

So the robust primary assertion is site-local: perturb the reserved input and assert the derived output value responds at the derivation's site, which is immune to the deadband false-positive because it observes the value, not its distant hash consequence. The whole-world-hash-moves check the gate described is a coarser proxy that happened to catch the soil case (a constant output also moves no hash) but would false-positive on a live-but-byte-neutral derivation. My recommendation is to make the site-local value-responds assertion the pass/fail liveness gate, and keep the run-path-hash-moves as a separate, softer coverage signal (informational, or gated only for derivations a scenario is expected to surface), never the liveness verdict. This is the one design decision I bring to the gate rather than settle myself.

## Substrate-first on values

The gate needs one class of value: the perturbation magnitude per input (the gate used 330x for the photosynthesis constants). It is reserved-with-basis, not fabricated: the basis is a factor large enough that a live derivation's output must move by more than the site's own rounding at its scale, read from the input's declared envelope rather than guessed, and small enough to stay within the quantity's representable range. Where the site-local assertion is used, the magnitude only has to exceed the derivation's output ULP, which the per-quantity scale already declares (R-UNITS-PIN), so no new owner value is owed; where the coverage signal is used, the magnitude is the reserved-with-basis sweep factor. Nothing is authored inline.

## The slice plan

Each slice self-audited against the section-9 lenses in one pass (cost directive: no spawned panels), byte-neutral where it can be (the gate is a test harness plus a data registry, off the canonical run path), gated by the gate.

1. **The registry substrate.** The `RetiredFloorDerivation` data row and its registry (id, derived-quantity handle, reserved input keys, scenario), plus the static cross-check that every `@derives:` site has a row and every row a site. Populate it from the seven current derivations as data. Byte-neutral (a new registry, unread by the run path).
2. **The site-local liveness probe.** The mechanism that, per registry entry, perturbs the declared reserved input and asserts the derived output value responds at its site, failing loud and naming the dead derivation. This is the pass/fail gate. Prove it fires: reproduce the `soil_baseline` dead case (a derivation wired to a constant) and show the probe fails, then show the live derivation passes.
3. **The coverage signal (softer, separate).** The run-path-hash-under-large-perturbation check as an informational coverage report, not a liveness verdict, so a live-but-byte-neutral derivation is reported as a coverage gap rather than failed. This is where the sigma-style honest limit is stated.
4. **Wire into CI and the harness.** Add the gate to the determinism harness and the CI determinism job, on the constructor-gate and determinism-gate pattern, with its own baseline where a ratchet fits.
