# Backing Architecture for a Custom Rust Emergent-Civilization Simulator: A Survey of Approaches

## TL;DR
- **Build a CPU-authoritative, deterministic ECS simulation with a sim-LOD tier system, and offload only environmental grid fields (water, heat, erosion, flow-field pathfinding) to the GPU via a single backend trait — but adopt CubeCL rather than hand-maintaining dual CUDA+Vulkan paths, because it JIT-compiles one Rust kernel to CUDA, ROCm/HIP, Metal, Vulkan/SPIR-V, WGSL and CPU-SIMD from one source.** This collapses the most expensive item in your plan (dual GPU backends) into a single maintainable codebase while preserving the CUDA-when-available / Vulkan-elsewhere behavior you want.
- **Your folklore/information-transmission requirement is the genuinely novel part and has direct academic precedent**: James Ryan's *Talk of the Town* (per-character "mental models" of "belief facets" carrying ground-truth-vs-belief, evidence, strength, provenance, and a "belief mutation graph") and Max Kreminski's *Gossamer* (witness→reflection→propagation→decay phases) give you a buildable architecture that layers cleanly on event sourcing — ground-truth events in a canonical log, per-entity lossy projections that distort at each transmission hop.
- **The DF bottleneck you should design around is NOT pathfinding** — Bay 12's own profiling shows pathfinding is only ~3–6% of CPU and that per-unit turn processing (over 60% in large forts) plus line-of-sight/proximity checks dominate; your sim-LOD + aggregate-pool design is the correct structural answer and matters far more than any single algorithm choice.

## Key Findings

1. **ECS choice:** For a relationship-heavy historical simulation, **flecs (via the `flecs_ecs` Rust bindings, Indra-db/Flecs-Rust)** is the strongest fit — it is, per its own documentation, the *"first open source ECS with full support for Entity Relationships"* (family trees, ownership, allegiance, artifact provenance as native graph edges). **Bevy ECS** is the best pure-Rust archetypal alternative with a mature parallel scheduler; **hecs** is the best minimal embeddable option for a fully custom engine. Sparse-set vs archetypal is a real tradeoff: archetypal (Bevy default, flecs, hecs) gives fastest iteration/SoA cache locality; sparse-set gives faster add/remove.

2. **GPU strategy:** Hand-written dual CUDA+Vulkan backends are a large, ongoing engineering cost. **CubeCL** (the compute layer under the Burn ML framework) eliminates most of it. If you want lower-level control, the clean abstraction is a `ComputeBackend` trait with a `wgpu` implementation (covers Vulkan/DX12/Metal/GL/WebGPU) plus an optional `cudarc` implementation for NVIDIA. `wgpu` alone satisfies "Vulkan elsewhere + Metal on macOS" with no extra macOS work because Naga translates WGSL→MSL natively.

3. **Determinism:** Keep canonical game state on CPU with integer/fixed-point math and per-entity counter-based RNG (PCG/SplitMix64/Philox) seeded by hashing `(master_seed, entity_id, phase)`. Treat GPU environmental fields as *approximate* and non-authoritative; never let them feed back into canonical state without quantization, because GPU floating-point reduction order and driver variance break bit-reproducibility.

4. **SIMD:** Use the `wide` crate for portable SIMD on stable Rust today; `std::simd` is still nightly-only in 2026. The historic AVX-512 downclocking penalty is an **Intel Skylake-X-era** concern only; Zen 4/Zen 5 have essentially no frequency penalty, so structure-of-arrays layouts that enable AVX-512 are safe to pursue on AMD.

5. **The information/folklore subsystem** is buildable today by combining event sourcing (canonical truth) with Talk-of-the-Town-style per-character belief facets and Gossamer-style four-phase gossip propagation, distorting per transmission hop, gated by a salience/familiarity score.

## Details

### 1. Entity Component System (ECS) — crate survey

The mandate is data-oriented design (SoA, cache locality) plus first-class relationships for family/ownership/allegiance/provenance graphs and LOD-tiered storage.

- **flecs / `flecs_ecs` (Indra-db/Flecs-Rust):** The standout. Flecs is, per its README, the *"first open source ECS with full support for Entity Relationships"* — relationships are native pairs `(Relationship, Target)`, e.g. `(ChildOf, parent)`, `(Owns, artifact)`, `(MemberOf, civ)`, with constant-time relationship queries, traversal, exclusive/union/oneof constraints, and reflexive relationships. This directly models lineage, who-killed-whom, artifact ownership, and allegiance without a separate graph database. The Rust API reports being "on par with C-level performance," is "stable and used in production" with a "1.0 release planned next quarter," and the docs note Flecs is "heavily tested, running more than 8000 tests in its core library alone and used in AAA engines. The Rust API itself has 500+ tests." Relationships were introduced to flecs in 2021. Flecs is written in C (FFI), is archetypal, and has a JSON reflection/serialization framework. Tradeoff: C dependency, FFI boundary, and a runtime-safety-lock cost (`flecs_safety_locks`, on by default) you can disable for release builds. Systems are single-threaded by default with opt-in pipeline parallelism.
- **Bevy ECS:** The most mature *pure-Rust* archetypal ECS, usable standalone (decoupled from Bevy-the-engine). Strengths: a sophisticated automatic parallel scheduler that infers data access from system signatures (`&` vs `&mut`) via `FixedBitSet` access masks and runs non-conflicting systems concurrently; per-component choice of `Table` (archetype, fast iteration) vs `SparseSet` (fast insert/remove) storage. ~46k GitHub stars, very active. Relationships are less first-class than flecs; for deep historical graphs you'd build adjacency yourself.
- **hecs:** "A fast, minimal, and ergonomic entity-component-system library." Archetypal, library-not-framework, no built-in scheduler — ideal if you want to own the engine loop entirely and bring your own Rayon parallelism. Best if "custom engine" is taken literally.
- **legion:** Historically fast archetypal ECS, but effectively unmaintained (as is Amethyst). Avoid for new work.
- **specs:** Popularized Rust ECS; parallel but legacy. Avoid for new work.
- **shipyard:** Active, EnTT-inspired (sparse-set lineage), good usability/speed. A reasonable pure-Rust alternative if you prefer sparse-set semantics.
- **Sparse-set vs archetypal:** Archetypal stores entities with identical component sets in contiguous tables → excellent SoA cache locality and iteration (what your SIMD/DOD mandate wants), but add/remove move entities between tables (graph-edge transitions). Sparse-set keeps per-component arrays with indirection → cheap add/remove, slightly slower iteration. For stable high-detail agents with bulk per-tick iteration, archetypal wins; reserve sparse-set storage for components that churn (transient status effects).

**Recommendation:** flecs_ecs as the primary store for its relationships, OR hecs + a hand-built relationship layer to avoid the C FFI. Use Bevy ECS if you want the scheduler for free.

### 2. Compute and multithreading

#### CPU parallelism frameworks
- **Rayon:** The default for CPU-bound data parallelism. Work-stealing scheduler over a global thread pool built on `crossbeam::deque`; `par_iter()` drop-in parallel iterators and a `join()` fork-join primitive. Right tool for "iterate all medium-LOD agents in parallel." Note meaningful overhead — one analysis found Rayon gave only ~2x speedup on 8 threads for a poorly-structured workload, and you sometimes need ~1e6 elements before Rayon beats single-threaded. Structure work into large chunks.
- **Tokio:** Async runtime for I/O-bound concurrency, NOT compute. Critical hazard: **do not mix Rayon and Tokio on the same threads.** Meilisearch's Louis Dureuill documented (blog "Mixing rayon and tokio for fun and (hair) loss") that `block_on` calls inside a Rayon job caused a panic/abort in Meilisearch v1.6; the fix (v1.6.1) was "not to: have them communicate via channels instead." Keep async I/O (save streaming, networking) on Tokio and simulation compute on Rayon with strictly isolated pools.
- **crossbeam:** Lock-free data structures, scoped threads, MPMC channels, and the work-stealing deque Rayon builds on. Use for inter-subsystem message passing and custom lock-free double-buffering.
- **std::thread + custom pools:** For a hand-rolled tick scheduler where you want full control of a tier-aware job graph, a custom pool over `std::thread` + crossbeam channels is viable.
- **enkiTS-style task graphs / Taskflow equivalents:** Rust lacks a single dominant enkiTS clone, but the building blocks exist: Rayon's `scope`/`join` for fork-join DAGs, or `bevy_tasks` (usable standalone) for a lighter scheduler. The pragmatic answer for a solo dev is Rayon scopes for the compute graph plus a hand-built phase scheduler.
- **Bevy's parallel ECS scheduler:** If you adopt Bevy ECS, you get automatic system-level parallelism — it analyzes each system's component access and runs systems with disjoint access concurrently, exactly the "lock-free parallel updates" pattern when combined with double-buffering.

#### Double-buffered world state
Maintain two copies of mutable world state (or per-tier hot component arrays); read from buffer A, write to buffer B, swap at tick boundary. This eliminates read-write contention and makes parallel agent updates trivially safe. Combine with command buffers (deferred structural changes applied single-threaded at the sync point) to avoid mid-tick mutation of the entity graph.

#### False sharing, cache-line alignment, NUMA
- **False sharing:** pad/align hot, independently-written per-thread data to 64-byte cache lines (`#[repr(align(64))]`) so two threads don't contend on one line. SoA layouts naturally help.
- **Cache-line alignment for SIMD:** align SoA arrays to 32 bytes (AVX2) or 64 bytes (AVX-512) for aligned loads.
- **NUMA / CCX/CCD topology:** On AMD Ryzen/Threadripper, cores group into CCDs/CCXs with separate L3 slices; cross-CCD traffic is slower, and multi-socket/Threadripper is true NUMA. Pin worker threads to cores and partition the map into regions affined to specific CCDs to keep a region's data L3-resident. Rust `core_affinity` and `hwloc`-based bindings provide pinning; Rayon lets you build a custom `ThreadPool` and pin its threads. **Caveat: I could not verify a specific current NUMA-aware Rust crate version in this pass; treat pinning as available via `core_affinity` and validate empirically on your AMD/Intel test hardware.**

#### SIMD options (detailed)
- **`std::simd` (portable SIMD):** In std but **nightly-only and expected to remain so for the foreseeable future** as of 2025–2026. Best platform coverage (everything LLVM supports), pairs with `multiversion` for runtime dispatch. Use only if you accept a nightly toolchain.
- **`wide`:** Mature, stable-Rust portable SIMD (wraps `safe_arch`); supports x86 SSE/AVX, ARM NEON, WASM. The pragmatic 2026 default. Downside: no built-in multiversioning (can't runtime-select AVX2 vs AVX-512 from one binary without `cargo-multivers` or manual dispatch), and thin docs.
- **`pulp`:** Built-in multiversioning, powers `faer` (proven performance). Limitation: operates on native SIMD width, so code must handle variable-width chunks.
- **`core::arch` intrinsics (AVX2/AVX-512, NEON):** Maximum control. As of Rust 1.87+, most x86 intrinsics are no longer `unsafe` to call, and `safe_unaligned_simd` covers the rest, so this path is far less painful than before. You hand-write per-ISA versions and do your own `is_x86_feature_detected!` dispatch.
- **Autovectorization:** With `-C target-cpu=native`, LLVM autovectorizes simple SoA loops; for a solo dev, well-structured SoA loops + autovectorization is the cheapest 80% solution.
- **ARM/portable bridge:** `wide` and `std::simd` both lower to NEON on ARM (Apple Silicon, ARM64 Linux), giving you a portable-SIMD bridge between x86 AVX and ARM NEON without per-arch code.

#### AVX-512 downclocking history (verified)
The "AVX-512 causes catastrophic downclocking" lore is **specific to Intel Skylake-X (2017) and similar pre-Ice Lake parts.** Skylake-X showed a ~50,000-cycle transition where 512-bit code ran at ~1/4 dispatch rate, then settled at reduced steady-state frequency (e.g., i9-10900X dropping from ~4.65 GHz scalar toward ~3.7–4.0 GHz). On **Zen 4** (double-pumped 256-bit) and especially **Zen 5** (native 512-bit datapaths, dual 512-bit L1D loads/cycle), there is essentially no frequency penalty — Zen 5 runs dependent AVX-512 at full ~5.7 GHz with at most a ~1.3µs blip, using di/dt adaptive clocking. As Alexander Yee concluded (numberworld.org, "Zen5's AVX512 Teardown," Aug 2024): *"every precaution and recommendation against AVX512 that has been built up over the years on Intel should be completely reversed on Zen5 (as well as Zen4). Do not hold back on AVX512."* Exception: Zen 5 mobile **Strix Point** retains the narrower Zen 4-style throughput and a nerfed vector unit vs desktop Granite Ridge. So: SoA layouts targeting AVX-512 are an unambiguous win on modern AMD desktop; on Intel, prefer 256-bit AVX2 unless you've confirmed Ice Lake+ behavior. Detect at runtime.

#### GPU compute for simulation

**The dual-backend question (central to your plan).** You plan CUDA (NVIDIA) + Vulkan-compute (everyone else). Findings:

- **CubeCL (tracel-ai) is the highest-leverage option and likely better than hand-maintaining two backends.** A single `#[cube]`-annotated Rust function JIT-compiles on demand to CUDA, ROCm/HIP, Metal, Vulkan/SPIR-V, WGSL, or CPU-SIMD, "while still using the best instructions each platform offers." It provides automatic vectorization, comptime specialization, autotune, and a memory-management strategy with heavy buffer reuse. It is the compute backend of the Burn ML framework (real production use). This collapses your dual-path maintenance into one kernel source while still hitting CUDA on NVIDIA and Vulkan elsewhere — precisely your goal. Caveats: it is a restricted DSL (functions, generics, structs; partial trait/method support), "still has rough edges," and is a young project from a small team; pin versions.
- **`wgpu` as the cross-platform compute abstraction:** Implements WebGPU over Vulkan (Linux/Windows/Android), Metal (macOS/iOS), DX12 (Windows), GL (fallback), and WebGPU/WebGL2 (browser). Shaders in WGSL, translated by **Naga** to SPIR-V/MSL/HLSL/GLSL. This single-handedly solves "Vulkan elsewhere + Metal on macOS" with no separate Metal path. Compute shaders are first-class; grid state stays in GPU-resident SSBOs across ticks. Caveats: WebGPU compute is a subset of raw Vulkan (no vendor extensions; subgroups/mesh shaders still maturing); perf varies across wgpu-native versions (one project saw an H100 regress ~20% between versions while an M3 improved ~20%).
- **`ash`:** Thin, unsafe, complete Vulkan bindings. Use for raw Vulkan control (subgroups, specific extensions) beyond WebGPU. Highest control, highest maintenance.
- **`vulkano`:** Higher-level, safer Vulkan wrapper. Middle ground, but Vulkan-only (still needs MoltenVK on macOS).
- **`erupt`:** Another Vulkan binding, less active than ash; ash is the safer ecosystem bet.
- **CUDA via Rust:**
  - **`cudarc`:** Actively maintained, works with CUDA 12.x and recent Rust; host-side kernel launching (PTX/cu). Community consensus favors it for practical NVIDIA work today. Limitation: does not share host/device data structures — you serialize to byte streams.
  - **Rust-CUDA project (`cust`, `rustc_codegen_nvvm`, `cuda_std`, `cudnn`):** Rebooted in 2025 after dormancy; brought to nightly-2025-03-02, CUDA 12.x support added, `cust` underwent its biggest-ever breaking release (primary-context handling, `cust_core` split out as `#![no_std]`). Lets you write kernels *in Rust* via NVVM IR. Promising but in active stabilization; Rust GPU is exploring merging Rust-CUDA and rust-gpu.
  - **rustc nvptx backend:** Experimental compiler backend emitting PTX; complementary to Rust-CUDA's NVVM approach.
- **rust-gpu:** Compiles Rust → SPIR-V for Vulkan-compatible workflows; graphics-oriented but supports compute. Pair with ash/wgpu/vulkano as host.
- **cuda-oxide:** Another rustc-codegen-to-NVIDIA effort; portable PTX output, launchable via cudarc.

**Clean backend abstraction.** Define a `ComputeBackend` trait with methods like `upload_buffer`, `dispatch(kernel_id, workgroups, bindings)`, `async_readback(slice)`, and resource handles. Provide a `WgpuBackend` (default, cross-platform) and an optional `CudaBackend` (cudarc) selected at runtime by capability probe — exactly the pattern the "Rust running on every GPU" demo uses (feature flags `wgpu`/`vulkan`/`ash`/`cuda`, multiple backends compilable into one binary, runtime selection; `wgpu,vulkan` forces MoltenVK on macOS). If you adopt CubeCL, this trait is largely subsumed by CubeCL's `Runtime`/`ComputeClient`/`ComputeServer`/`Channel` abstraction, which already wraps the wgpu and cuda runtimes.

**Which workloads belong on GPU.** All are grid/stencil/SIMD-friendly:
- Water flow/pressure, heat/fire/smoke diffusion, hydraulic & thermal erosion, rainfall/wind, soil fertility, pollution/disease-as-diffusion, magic fields — classic CA/PDE stencils. The race-condition pitfall (two cells moving into one) is solved with **block cellular automata** (Margolus neighborhood: 2×2 blocks shifted diagonally on alternate timesteps), the standard GPU falling-sand/water technique. Shallow-water-equation finite-volume solvers are routinely GPU-parallelized on uniform grids.
- Procedural terrain genesis (simplex/worley noise, hydraulic erosion, river carving) — embarrassingly parallel per-cell.
- **Flow-field / eikonal pathfinding** — parallel wavefront Dijkstra/eikonal solves produce a vector field thousands of agents sample for free, the alternative to per-agent A*.

**Why agent decision-making stays on CPU.** Agent brains are branchy, divergent, and pointer-chase the social graph. GPU warps/wavefronts execute in lockstep (SIMT); divergent control flow serializes (each branch path executed by the whole warp with masking), destroying throughput, and irregular graph walks defeat coalesced memory access. So canonical agent AI stays CPU-side; the GPU owns regular grid fields.

**The CPU↔GPU seam (critical).** PCIe has high latency and limited bandwidth relative to on-GPU memory. Rules:
- Keep grid state **resident in GPU buffers (SSBOs/structured buffers) across ticks** — never round-trip the whole grid each frame.
- Read back **only the slices CPU agent-brains need**, using **asynchronous double/triple-buffered readback** (map a staging buffer, read last frame's copy while the GPU computes the next).
- **Never** do synchronous readback or per-agent-per-tick small transfers — latency dominates and stalls both processors. Batch all agent queries into one readback region.

### 3. Determinism

- **Per-entity RNG streams:** Counter-based/splittable generators are ideal — no shared state, reproducible regardless of execution order. SplitMix64 ("9 operations per 64 bits," loop/branch-free, explicitly "suitable for SIMD or GPU implementation"), PCG (NumPy's default), and Philox (counter-based, the recommended choice for "GPU or massively parallel computing") are all good. Seed each stream via **hierarchical seed splitting**: `S_k = hash(k, parent_seed)`, giving deterministic per-entity, per-phase substreams from one global seed without tracking draw counts. Your proposed `master_seed XOR entity_id` works, but **use a hash (e.g., a SplitMix64 finalizer), not XOR**, to mix bits and avoid collisions between correlated/nearby IDs.
- **Parallel floating-point reduction hazard:** FP addition is non-associative, so a parallel reduction (sum over agents, Rayon `reduce`) can give different results run-to-run depending on chunk/thread scheduling. This is the single most common source of nondeterminism in parallel sims. Mitigations: (a) use **integer/fixed-point** for any canonical accumulator; (b) force deterministic reduction order (sequential fold or a fixed tree order independent of thread count); (c) compile with explicit FMA (`std::fma`) to avoid compiler-dependent contraction. HPC literature confirms only serial or fixed-order executions are bit-reproducible.
- **Fixed-point/integer canonical state:** Make authoritative state integer or fixed-point. This guarantees cross-platform, cross-run bit-identical worlds from a seed — exactly DF's property (same seed → same world on any machine, since DF worldgen uses a seeded PRNG).
- **GPU non-determinism tension:** GPU results vary by driver, hardware, and reduction order. **Resolution: keep canonical state deterministic on CPU; let the GPU own approximate environmental fields.** When a GPU field must influence canonical state, **quantize** it (e.g., snap water depth to integer millimeters) before it crosses into authoritative state, so small FP variance can't diverge the canonical timeline. Treat GPU fields like a renderer's approximation, not the source of truth.

### 4. Entity / individual modeling (personality, goals, AI at scale)

**Personality/value models:**
- **OCEAN/Big Five numeric vectors:** Compact (5 floats), SoA-friendly, easy as utility-curve inputs. Talk of the Town uses extroversion as a numeric personality component driving conversation volume — proof numeric trait vectors integrate cleanly with social sim.
- **DF facet/value/goal system:** DF models ~50 personality facets, cultural/personal values, and long-term goals; richer and more legible in generated text ("she valued craftsmanship and loathed cowards"). Store as small fixed arrays per high-LOD entity.
- **Decision-making architectures (map onto ECS + LOD):**
  - **Utility AI (need-based):** Each candidate action scored 0–1 by curves over agent/world state; pick highest. Scales well, smooth behavior, no manual priority encoding, naturally data-oriented (score arrays over SoA). A practitioner comparison found Utility AI fastest to implement (~4h vs 8h for BTs, 12h for GOAP) with better target-selection behavior. **Best default for thousands of agents.**
  - **GOAP:** A*-over-actions to assemble multi-step plans (F.E.A.R. lineage). Emergent plans, but high per-agent planning cost — reserve for high-LOD, narratively load-bearing agents.
  - **HTN:** Scales better than flat GOAP because methods constrain search (Killzone/Horizon lineage). Good for structured civ/economic behaviors.
  - **Behavior trees:** Great for authored, legible reactive behavior; become unwieldy as complexity grows.
  - **Hybrids:** Production AI usually mixes — utility-scored selector nodes in a BT, or GOAP actions executing as BT subtrees ("Goal-Oriented Behavior Tree").
- **LOD mapping:** High-LOD → full utility AI / GOAP / HTN per tick. Medium-LOD → coarse utility every N ticks. Aggregate pools → no per-agent AI; stochastic event generation over population statistics.

### 5. Event-sourced history + folklore (the distinctive subsystem)

**Event sourcing foundation.** Maintain an append-only event log; every event carries actor IDs, location, timestamp (tick/date), and tagged subject entity IDs (artifact ID, building ID, victim ID). An entity's life history, an artifact's provenance, a building's history, and who-killed-whom are all *queries over the log filtered by entity ID*. Lineage/family trees and ownership/allegiance graphs are best modeled as **flecs relationships** (native graph edges) layered over the log. Tarn Adams describes DF history exactly this way: "a giant zero-player strategy game... and history is just a record of that," noting "procedurally generating stories by recording a log of a simulation is a valid enough approach, though it has drawbacks... you need to do post-processing or investigation to find any good moments."

**Persistence / save bloat.** DF saves balloon because full records are retained for every historical figure and event across centuries. Policy: retain full append-only records **only for promoted (high-LOD) entities**; aggregate pools persist as compact statistics; use **log compaction** (snapshot + truncate old fine-grained events into summaries) for demoted entities. Binary serialization:
- **rkyv** — zero-copy deserialization; fastest at load. As David Koloski's benchmark concluded, rkyv "won nearly every performance category, and was highly competitive with the winner when it didn't... without compromising on size" (current `rust_serialization_benchmark` shows rkyv 0.8 archived-access at ~1.245ns). Ideal for the huge mmap-able world snapshot.
- **bincode** — fastest *serialize*, compact, simple; good for the streaming event log.
- **postcard** — like bincode but smaller output (good for size-sensitive logs).
- **bitcode** — strong size+speed balance.
- Avoid JSON for canonical state; reserve for human-readable Legends-style exports.

**Information transfer / folklore formation — academic grounding and architecture.** This is the deepest requirement, with direct, buildable precedent.

*Cultural-transmission science (what to model).* Transmission-chain ("telephone game") experiments since Bartlett (1932) consistently show material gets (1) shorter and (2) loses detail across hops, and is distorted toward the receiver's pre-existing schemas/stereotypes (assimilation), with generalization and abstraction. Content biases mean negative, threat-related, disgust-related, and socially salient information transmit better. The **accumulated copying-error model** (Eerkens & Lipo) gives a quantitative null model: independent transmission chains diverge over time (between-chain variance grows) while the mean stays constant, because each copy adds small random error. These give you the *distortion functions*: per-hop truncation, detail loss, schema-conforming mutation, and content-bias-weighted survival.

*Computational precedent (how to build it).* Two academic systems give a concrete architecture that layers directly on event sourcing:

- **James Ryan & Michael Mateas, "Simulating Character Knowledge Phenomena in *Talk of the Town*"** (Game AI Pro 3, ch. 37, 2017; and "Toward Characters Who Observe, Tell, Misremember, and Lie," EXAG 2015; framework in Python). Each character holds an *ontology of mental models*, one per person/place, each a list of **belief facets** corresponding to attributes. Crucially, each attribute has a **ground-truth value**, but a belief facet "will represent a character's potentially false belief about that attribute." Each facet stores: **value**; **predecessor** (prior belief — a perfect history of the character's beliefs); **parents** (the source character's facets that spawned this belief — full provenance, letting the system "trace the history and trajectory of any piece of information"); **evidence** (accumulating list); **strength** (sum of evidence strengths, decays with time); **accuracy** (vs current truth). They model **eleven knowledge phenomena across five categories**: origination (reflection, observation, transference, confabulation, lie, implant), reinforcement (declaration — retelling strengthens belief, so "an NPC who frequently tells the same lie might come to actually believe it"), propagation (statement, eavesdropping), deterioration (mutation), termination (forgetting). **Mutation** uses an authored **belief mutation graph** — a value→value transition-probability table (e.g., hair "brown" → black/red more probable than white). Per the chapter, facet deterioration probabilities are "determined by the character's memory attribute (modeled as a floating-point value inherited from a parent), the facet type (e.g., a whereabouts belief will be more likely to mutate than a first name belief), and the strength of the existing belief (weaker beliefs are more likely to deteriorate)." **Confabulation** samples from the town's actual attribute distribution (if 45% have black hair, a confabulated hair belief is 45% likely to be "black"). Scale: 300–500 NPCs, each holding 250–600 mental models and ~800–1200 belief facets; one knowledge timestep takes ~a minute (explicitly "not very computationally efficient"). Used in the live game **Bad News** (Samuel et al. 2016).

- **Max Kreminski, "Toward Better Gossip Simulation in Emergent Narrative Systems"** (Gossamer, IEEE CoG 2023; open-source JS, github.com/mkremins/gossamer). Argues a full gossip simulation needs **four phases**: **witness** (determine who saw an action, possibly mutate their perception "allowing for possible misunderstanding," decide whether/how strongly to record it), **reflection** (assemble remembered actions into communicable "microstories"), **propagation** (decide which microstories to share, "possibly mutate the action memories underlying shared stories in their transmission," set the recipient's initial belief strength), and **decay** (reduce memory weights over time, cull below threshold — prioritizing recent events and bounding memory growth). Each character has a per-character memory database (a DataScript DB, queried via the **Felt** story-sifting language). Distortion happens at both the witness and propagation phases — *every witness and every relay is a potential distortion point.* Probability of faithful recording/retention/transmission is gated by **salience**, modeled as **familiarity** = normalized count of references to an entity within the character's memory DB. (Note: Gossamer is a prototype; Kreminski explicitly states it was "not yet" evaluated, and the emergent behaviors he lists — reputations preceding characters, dueling rumors, "famous for being famous" — are stated as hoped-for targets, not demonstrated results.)

*How to architect on top of event sourcing.* The two systems map cleanly onto your plan:
1. **Ground truth = canonical event log** (CPU, deterministic). Each event immutable.
2. **Per-entity belief = a lossy projection** of events that entity has evidence for (belief facets / per-character memory DB). An individual's belief about a past event can therefore differ from ground truth — satisfying your requirement directly.
3. **Distortion accumulates per transmission hop:** when A relays to B, apply the per-hop transform (truncate, drop detail, mutate values via the belief mutation graph, weighted by content bias and salience). After K hops / G generations the belief drifts measurably — folklore/myth/legend formation falls out of accumulated copying error.
4. **Provenance is preserved** via parents/predecessor pointers so you can reconstruct the *true* trajectory of a rumor even after mutation — perfect for a Legends-mode view showing both "what really happened" and "what people believe happened."
5. **Lies** are injected events whose payload diverges from ground truth while source/time metadata is preserved (Talk of the Town's "false flag").
6. **Belief strength + decay** gives confidence-weighted, time-decaying beliefs with possible oscillation when contradictory evidence arrives.

*Other emergent-narrative systems worth studying:* **Versu** (social-practice-driven characters), **Ceptre** (linear-logic rule language for simulating interactive systems/narrative), **Neighborly** (an open re-implementation of Talk of the Town as a reusable social-sim engine), and **Bad News** (the playable demonstration of Talk of the Town's knowledge model). DF's own rumor system is the commercial state of the art: rumors originate with witnesses of a historical event, spread person-to-person, decay over weeks/years while keeping longer-term reputation effects, and are tracked per-historical-figure with timestamps at individual/site/civ levels. Notably, DF currently spreads **no false rumors except about secret identities**, and (per Tarn Adams) does **not** model memory fallibility — precisely the gap Talk of the Town fills and where your sim can exceed DF.

### 6. Spatial data structures for a huge map
- **Tiling/chunking:** Fixed-size chunks; load/simulate/persist per chunk. Foundation for LOD and parallelism (each chunk an independent work unit affined to a thread/CCD).
- **Quadtrees/octrees:** Hierarchical subdivision for variable-density queries and multi-resolution LOD (coarse at top, fine at active sites). Octree for 3D z-levels.
- **Spatial hashing:** O(1) average neighbor queries; cheap per-tick updates; ideal for the line-of-sight/proximity checks that actually dominate DF's CPU (§7).
- **Regional partitioning:** Group chunks into regions for coarse simulation, parallel dispatch, and the high/medium/aggregate LOD boundary.

### 7. Pathfinding at scale
**Reframe the problem first.** Bay 12 dev profiling (cited via the DF community and wiki) shows FPS death is **not** primarily pathfinding. DF dev Putnam stated: *"Pathfinding happens so rarely that my profiling has shown it contributes to about 6% of an individual tick's processing time,"* and *"in a typical large fort, pathfinding consumes 3-5% of CPU time, based on profiling data."* The DF wiki's Maximizing Framerate page states: *"The vast majority of processing time in Dwarf Fortress is taken up by units taking their turns, over 60% in larger forts, of which less than 10% is actually pathfinding related."* Per-unit line-of-sight/proximity checks are the bigger algorithmic share (~20%), and Putnam specifically optimized line-of-sight code for v50.09 because profiling showed it was the bottleneck. **The structural fix is your sim-LOD + aggregate-pool design (fewer active per-tick units), not a fancier path algorithm.** That said:
- **Flow fields:** One Dijkstra/eikonal solve per destination yields a vector field any number of agents sample for O(1) per-agent steering — superb when many agents share goals; GPU-parallelizable. Downsides: high memory and slower-than-A* for a *single* agent on large maps; congestion/lane-forming artifacts need steering behaviors.
- **HPA\* (hierarchical A*):** Precompute a coarse graph of region portals; path at the abstract level then refine. Scales to huge, changing worlds — the canonical big-map answer. Combine with flow fields (Game AI Pro, "Crowd Pathfinding and Steering Using Flow Field Tiles": HPA* for the global route, flow-field tiles for local crowd movement).
- **Navmesh vs grid:** Navmeshes give cheaper long paths over open terrain but are costlier to update under frequent digging/construction; grids are simpler and update trivially (DF is grid-based). For a mutable fortress, grid + HPA* hierarchy is pragmatic.
- **Hierarchical multi-robot/MAPF research** reports ~500× speedups by decomposing into cells with parallel intra-cell solves — relevant if you ever need true collision-free multi-agent paths.

### 8. LOD promotion/demotion machinery
- **Promotion (anonymous → full entity):** When an aggregate-pool individual becomes narratively load-bearing (kills a noble, forges an artifact), instantiate a full ECS entity, generate a name/personality/backstory consistent with the pool's statistics and the triggering event, and **persist** it. Generation must be deterministic from `(pool_seed, slot_id)` so re-promotion is reproducible.
- **Demotion (full → summary):** Summarize the entity into the aggregate pool while **retaining its recorded history** — write a compacted life-summary record and key relationships into the persistent log, then free the hot components. The append-only history is never deleted, only the live representation.
- **Coherence:** Maintain a stable generational-index ID (§10) that survives promotion/demotion so log references and relationship edges stay valid. The aggregate pool tracks counts/distributions; promotion decrements the pool and creates an entity with a back-reference to it.

### 9. Worldgen pipeline
Mirror DF's structure: (1) **terrain genesis** (GPU noise → elevation, then rainfall/temperature/drainage/volcanism fields, biome classification — DF derives biomes from elevation/temperature/rainfall/drainage); (2) **erosion + hydrology** (GPU hydraulic/thermal erosion, river carving from mountains to oceans/lakes); (3) **history simulation** — the "giant zero-player strategy game with thousands of agents and loose turn rules" running at aggregate/abstract LOD: place civilizations and beasts, simulate expansion, wars, migrations, artifact creation, and notable figures, **recording every event into the event log.** This is the abstract statistical pass; detailed local sim begins only where the observer focuses. Determinism (§3) makes the whole pipeline reproducible from a seed, exactly like DF's seed→identical-world property.

### 10. Memory architecture and persistence
- **Arena/slab allocation:** Allocate hot component arrays in arenas/slabs for contiguous SoA layout and cheap bulk free. Per-chunk arenas align with spatial partitioning and NUMA affinity.
- **Generational indices for entity IDs:** `(index, generation)` so freed slots reuse without dangling references — stale handles fail the generation check. The standard Rust ECS pattern (flecs/hecs/Bevy all use it internally). Essential for LOD promotion/demotion churn.
- **Save formats:** rkyv for the mmap-able world snapshot (zero-copy load), bincode/postcard for the streaming event log (§5).
- **Save bloat for centuries of history:** full records only for promoted entities; compaction/snapshotting for the rest; aggregate pools stored as statistics, not individuals.

### 11. Procedural narrative generation
Turn the event log into readable stories by querying events by entity ID and rendering through templates/grammars. For an artifact: select all events tagged with its ID chronologically ("forged by X in year N, wielded by Y who slew Z, lost in the sack of W"). The challenge (per Tarn Adams and the Talk of the Town team) is **story sifting** — surfacing the *interesting* subsequences from a vast boring log. Tools: Kreminski's **Felt** story-sifting language and **Centrifuge** sifting tool; template/grammar realization (Tracery-style) over the queried sequence. Generate "the same gossip causes different reactions in different recipients" by realizing from each character's *belief* projection rather than ground truth.

### 12. Rendering alongside GPU compute
- **wgpu** is the natural fit: one device shared between compute and render, minimal 2D/tile rendering for basic art. Since the GPU is busy with simulation compute, budget GPU time and consider rendering at a lower tick rate than simulation.
- **Headless option:** Since "for now gameplay is just watching" and art is minimal, run **headless** (no render) for fast worldgen/history simulation and attach a thin tile renderer only for observation. This frees the entire GPU for compute during long runs.
- **Sharing GPU between compute and render:** schedule compute dispatches and render passes on the same queue/device; keep grid buffers GPU-resident and let the renderer read them directly (no CPU round-trip) when visualizing water/heat/etc.
- Other Rust render options: raw ash/Vulkan (overkill here) or a 2D crate, but wgpu's cross-platform coverage (incl. Metal on macOS via Naga) makes it the clear default. Note macOS has no native Vulkan and no CUDA — wgpu's Metal backend (or MoltenVK under wgpu/ash) is the only sane macOS path, which is another reason a wgpu-centric or CubeCL compute layer beats a hand-rolled CUDA+Vulkan plan for cross-OS reach.

## Recommendations

**Stage 0 — Spike the risky seam first (1–2 weeks).** Before committing, prototype the CPU↔GPU seam: a GPU water/heat CA in **CubeCL** (or wgpu compute for lower-level control), grid-resident across ticks, with async double-buffered readback of one slice. Measure readback latency and confirm you can keep canonical CPU state deterministic while the GPU field stays approximate. **Threshold to proceed with GPU offload:** if async readback of the agent-relevant slice costs under ~10–15% of your tick budget, GPU offload is worth it; if synchronous stalls dominate, keep environmental sim on CPU with SIMD initially.

**Stage 1 — Deterministic CPU core.** Stand up the ECS (**flecs_ecs** for relationships, or **hecs** for zero FFI + a fully custom loop). Implement integer/fixed-point canonical state, per-entity counter-based RNG via hierarchical seed splitting (hash, not XOR), double-buffered world state, and a Rayon-based tick scheduler with isolated thread pools (no Tokio on compute threads). Build the append-only event log + rkyv snapshot / bincode log persistence now — retrofitting event sourcing later is painful.

**Stage 2 — Sim-LOD + worldgen.** Implement the three-tier LOD with promotion/demotion over generational-index IDs, then the DF-style abstract history pass (aggregate "zero-player strategy game") recording all events. Validate that the same seed yields a bit-identical world across machines — your determinism acceptance test.

**Stage 3 — Agents and the folklore subsystem.** Add utility AI for high-LOD agents (cheapest path to good behavior at scale; add GOAP/HTN only for the few narratively critical agents). Then build the information/folklore layer: per-entity belief projections (Talk-of-the-Town belief facets with ground-truth-vs-belief, evidence, strength, provenance pointers), Gossamer's four-phase witness/reflection/propagation/decay loop, an authored belief-mutation graph, and salience-gated distortion per hop. This is your differentiator from DF (which lacks memory fallibility) — invest here, but keep full belief-modeling restricted to promoted entities (Talk of the Town is ~1 min/timestep for only 300–500 NPCs), with aggregate statistical rumor-spread for the masses.

**Stage 4 — Pathfinding and spatial scaling.** Implement spatial hashing for the proximity/LOS checks that actually dominate (not pathfinding), chunk/region partitioning with NUMA-affined worker pools, and HPA* + flow-field tiles for movement. **Benchmark trigger:** profile before optimizing pathfinding — if it's <10% of tick time (as in DF), spend effort on per-unit turn processing and LOS instead.

**Stage 5 — SIMD, NUMA, and rendering polish.** Convert hot SoA loops to `wide` (stable) SIMD; target AVX-512 width on AMD Zen 4/5, AVX2 on Intel pre-Ice-Lake (detect at runtime). Pin threads to cores/CCDs via `core_affinity`. Add the thin wgpu tile renderer (or stay headless for batch worldgen).

**Where your stated plan should change:**
- **Replace hand-maintained dual CUDA+Vulkan with CubeCL** (or a wgpu-primary backend + optional cudarc), unless you specifically need raw CUDA/Vulkan extensions. The single biggest effort-saver; still gives CUDA-on-NVIDIA / Vulkan-elsewhere and solves macOS (no native Vulkan/CUDA) for free.
- **Re-prioritize away from pathfinding** as the feared bottleneck; design for per-unit turn-processing and LOS scaling instead.
- **Use a hash, not XOR, for per-entity seeding**, to avoid correlations between nearby entity IDs.

## Caveats
- **Source/verification gaps:** I could not verify a current, specifically-versioned NUMA-aware Rust crate; thread pinning via `core_affinity` is available but validate topology behavior empirically on your AMD/Intel hardware. Several Rust GPU/ECS projects (Rust-CUDA, CubeCL, flecs_ecs, rust-gpu) are young and fast-moving — pin versions and expect breaking changes; CubeCL and Rust-CUDA saw heavy 2025 churn.
- **Gossamer is a prototype, not a validated system:** its author explicitly states it was not yet evaluated, and the impressive emergent behaviors are goals, not demonstrated outcomes. Talk of the Town is more proven (shipped in Bad News) but explicitly "not very computationally efficient" at ~1 min per knowledge timestep for 300–500 NPCs — your LOD system must restrict full belief-modeling to promoted entities.
- **DF FPS-death attribution** comes from community-reported Bay 12 developer profiling (forum posts, DF wiki, dev Putnam's statements), not a formal publication; the ~3–6% pathfinding / ~20% LOS / >60% unit-turn figures are from profiling of specific forts and may vary by fortress.
- **wgpu compute performance varies across versions and backends** (documented ~20% swings between wgpu-native releases, MoltenVK-on-macOS overhead); benchmark on each target platform rather than assuming portability implies uniform performance.
- **AVX-512 guidance is microarchitecture-specific:** the "no penalty" conclusion holds for Zen 4/5 desktop (Granite Ridge) and Ice Lake+ Intel, but NOT Skylake-X-era Intel or Zen 5 mobile Strix Point. Detect at runtime.
- **Determinism is fragile under parallelism:** any stray parallel FP reduction, hash-map iteration order, or thread-count-dependent work split can break reproducibility. Budget for a determinism test harness (same seed, different thread counts → bit-identical state) from day one.
