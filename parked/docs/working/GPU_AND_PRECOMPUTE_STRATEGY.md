# GPU and precompute strategy (from the Fable Max consult, plus the derived-cache pass)

This file captures two related optimization directions for the end-of-buildout perf phase: a GPU-shader viability verdict on the engine's heavy kernels (a Fable 5 Max advisor consult, run on a fully de-domained code packet so the biology-guardrailed model saw only computational structure), and the owner's run-invariant precompute-cache ("shader cache") pass. Both are profile-gated: nothing here is built speculatively, and the earlier perf phase already showed a plausible target can be dead under 2 percent while an unglamorous allocation fix is 35 percent.

The consult's raw verdict lives in ephemeral job scratch; this is the durable, de-genericized record. The kernels were presented to the advisor as K1 through K15 with all domain identifiers stripped; the mapping back to the real code is below.

## The determinism reframe (the load-bearing insight)

The engine's bit-exact reproducibility criterion is SEMANTICS-equivalence, not implementation-equivalence. Every operation on the fixed-point path is pure integer arithmetic with a fully specified result (truncating division, floor shift, wrapping narrow), so any correct lowering on any hardware produces identical bits. GPU determinism can only break in two places: order-dependent cross-thread combination (a non-associative reduction), and accidentally changing an operation's semantics in the port (rounding direction, shift kind, overflow path). This is why fixed-point ports are far safer than float ports: integer sum and max are exactly associative and commutative, so a GPU tree reduction in any order at any block size is bit-identical and worker-count-invariant by construction.

The design line to hold: the day a kernel needs agents scattering into shared cells via atomics is the day worker-count invariance is at risk. Every kernel surveyed stays on the safe side of that; keep it there.

## Kernel map and per-kernel verdict

| Packet id | Real kernel | Verdict |
|---|---|---|
| K1 | `runner.rs` `Field::step` (diffusion + relaxation stencil) | CLEAR WIN. Canonical GPU kernel, one thread per cell (not per row), branchless clamped-edge index, double-buffered. Compute-dense with the sub-limb multiply, so it favors the GPU even at moderate grids. Already proven bit-identical on the 5090. Split the optional-scale branch into two comptime kernel variants. |
| K2 | `runner.rs` `Field::gradient_at` | Device function, not a launch. Fold into whatever per-agent kernel consumes it; keep as the host query API on CPU. |
| K3 | `stocks.rs` `Stock::regen_increment` / `Stock::step` (logistic step) | Device function, fuse into K4/K5. CAUTION: the port must mirror both the divisor-zero and the checked-overflow-to-ZERO paths (see the K11 finding). |
| K4 | `homeostasis.rs` `metabolize_derived` + `gpu/being.rs` flat-index form | CLEAR WIN, the best-shaped kernel. Consecutive index walks channels within an agent so the per-element reads coalesce, the per-agent input broadcasts, the small periodic arrays cache; fully independent, one real division per element for arithmetic weight. Worth it once the element count reaches tens of thousands. |
| K5 | `environ.rs` `regrow_supply` (grid regrow with floor injection) over the ResourceField | WIN at grid scale. Elementwise, no neighbors (serial only by accident). Co-ticks with K1 over same-sized grids, so at minimum share residency; fusing K1 and K5 saves a full grid read-write. The floor branch is exactly `v = floor`, write it that way. |
| K6 | `core/fixed.rs` `Fixed::exp` (18-term Horner series) | Device function. KEY OPTIMIZATION: the per-step `div(from_int(i))` is a division by a compile-time integer constant, which reduces algebraically to native truncating i64 division (the 2^32 scale cancels); do NOT route it through the 96-step K11, that is an order-of-magnitude cost for nothing. `scale_pow2`'s wide-overflow check rewrites as an i64 threshold compare. |
| K7 | `core/fixed.rs` `Fixed::ln` (13-term series) | Device function. `ilog2` maps to the GPU find-MSB (verify CubeCL's matches Rust's over the input range); the 13 Horner constants are compile-time, precompute them into a const table for zero in-loop divisions, leaving one real division. |
| K8 | `core/fixed.rs` `cordic_rotation` / `sin_cos` | Device function, textbook GPU CORDIC (32 fixed iterations, const table, select-ify the per-iteration sign branch). SHARP FLAG: `bits() >> i` on a negative i64 is an ARITHMETIC shift in Rust and must lower to an arithmetic shift, not logical; CUDA is fine, but pin a negative-input test vector so a future retarget cannot silently diverge. |
| K9 | `core/fixed.rs` mul/div/checked_mul/saturating_add/sqrt (i128 primitives) | STAYS ON CPU as the oracle the shader forms are proven against. There is no i128 through CubeCL, which is why K10 through K12 exist. |
| K10 | `gpu/prim.rs` `q32_mul` (u16 sub-limb multiply) | Substrate, inlined everywhere. Keep the regression vector on the negate-before-extract path (negative product, nonzero discarded bits, i64::MIN magnitude): negation over the full 96-bit value before extracting bits 32..95 is what makes negative-product floor-rounding and overflow-wrap match K9; a "simplification" that skips the low word breaks rounding. On CUDA specifically, a u32-limb form using a 32x32-to-64 widening multiply cuts partial products from 16 to 4 and stays exact. |
| K11 | `gpu/prim.rs` `q32_div` (96-step restoring division) | Substrate, and the packet's single cost center (~2000 instructions per division). REAL FINDING: it has a nonzero-divisor precondition and silently ignores its overflow limb, while K3's CPU path maps overflow to ZERO, so a port would silently diverge on tiny caps. Add quotient-overflow detection (test the high limb and the sign bound, take the ZERO path). A float-reciprocal-estimate-plus-exact-integer-fixup form is the known-fast alternative (floats only in the estimate, the fixup converges to the one specified truncated quotient, so bit-exactness survives), but only if profiles show division dominating and the fixup is proven exhaustive first. |
| K12 | `gpu/prim.rs` `isqrt_u96` (48-step bit-by-bit) | Substrate, inlined; between K10 and K11 in cost. |
| K13 | `runner.rs` `unit` (2D normalize) | Device function. Cost is one sqrt plus two divides per element; fine on GPU, marginal standalone (agent counts in the low tens of thousands finish in microseconds either way). Do NOT rewrite to one reciprocal and two multiplies, it changes rounding. |
| K14 | `locomotion.rs` `perceive` (per-agent window gather) | MARGINAL, and the constraint is structure not arithmetic. Port as one thread per agent walking the window in the CPU's exact order into a private per-agent slot range; do NOT parallelize over window cells within an agent, because the append order would then depend on scheduling. Needs SoA flattening; reads are uncoalesced. Worth it only with full residency and a device-side consumer. |
| K15 | `evolve.rs` per-generation scoring map plus sort | SPLIT IT. The map (N independent heavy rollouts) is potentially the biggest single win, the classic GPU batch-rollout, if the per-item evaluation is CubeCL-expressible (integer RNG ports exactly; watch register pressure and divergence). The sum and max reductions may go device-side freely (integer, order-invariant). The sort's index tie-break makes it a strict total order so any correct sort reproduces the permutation, but unless N is large the sort is host-side microseconds. Pragmatic split: GPU returns the metric array, host does the sum/max/sort/select tail. |

## The winning architecture (cross-cutting)

Per-kernel offload is the wrong unit. K1, K4, K5, and K14 all touch the same world state every tick. The architecture that wins is a resident world-state buffer set uploaded once, a fixed ordered launch graph per tick (K1 stencil, then K5 regrow, then K4 draw, then K14 gather), snapshots downloaded rarely, and many ticks (or many batched worlds, which amortizes launch overhead for small grids) between host round-trips. This matches the residency and batch-worlds GPU strategy from the earlier perf work.

## Three concrete follow-ups (low-risk, do regardless of the larger GPU pass)

1. Add quotient-overflow detection to `gpu/prim.rs` `q32_div` (K11) so the checked-to-ZERO semantics of `Stock` (K3) and the metabolize update (K4) survive a GPU port. This is a real latent divergence, worth fixing on its own.
2. Route the compile-time constant divisions in `Fixed::exp` and `Fixed::ln` (K6/K7) through native i64 constant division rather than the 96-step divider. Order-of-magnitude saving in the hottest transcendental paths, exact by definition.
3. Pin regression vectors on `gpu/prim.rs` `q32_mul` (K10) negate-before-extract floor-rounding, and on the CORDIC/exp arithmetic-shift-on-negative path (K8/K6), so a future backend retarget cannot silently change rounding or shift kind.

## The run-invariant precompute pass (the "shader cache", owner direction)

A planned end-of-buildout, profile-gated pass: at world-init / calibration time, precompute the derived values that the per-tick path recomputes repeatedly but that are invariant over the run (functions of the fixed calibration inputs and per-world data, not per-tick state), and have the tick path read the cache instead of recomputing.

Constraints that make it safe and principled:
- BYTE-NEUTRAL by construction, as memoization not approximation: the cached value must be the bit-exact result of the same fixed-point computation the live path would run (same ops, same order). The test is `cache == live recompute` bit-for-bit; the moment a precompute rounds or reorders differently, the pins move.
- THREE-WAY-TEST aligned: a cached derived value is a category-3 (derivable) quantity materialized once, exactly the pattern `time.life_cadence_ticks` already follows (derived in `clock.rs`, not authored). The pass is "materialize every run-invariant derivation at init," systematically, and it stays on the right side of the value-authoring line.
- PROFILE-GATED: only cache recomputations that profiling shows are both hot and run-invariant. This is a natural profiling workflow when the time comes: fan out profilers over the tick path, rank the hot run-invariant recomputations, and derive the cache set from the ranking rather than guessing.

This is the run-level scale of the same hoist-the-invariant principle the GPU verdict applies at the instruction level (K6/K7 constant-folding) and the tick level (resident buffers, many ticks per round-trip).

## The Fable pipeline consult: design, hardware validation, and measured speedup (2026-07-10)

A second Fable-5-Max advisor consult (routing confirmed Fable throughout, run on the de-domained current-GPU-code packet) designed the pipeline and the caching foundation, an Opus executor ran the tests on the RTX 5090, and Fable finalized an implementation-ready handoff. The raw handoff is in the run's `fable_pipeline.md`; the durable substance follows.

**Current-code diagnosis.** Every launcher is a one-shot: upload inputs, launch, read back (device sync plus D2H), return a Vec. There is no residency across a tick, and even the one residency-aware type re-uploads the whole field every dispatch. The transfer-per-kernel-per-tick is the dominant cost.

**Hardware validation (RTX 5090, CUDA): all 8 equivalence tests pass bit-for-bit, zero mismatches.** exp native-division (68,011 inputs), ln const-coefficient (22,191), div_checked overflow parity (40,054 pairs, 50 genuine overflows all returning 0 to match the CPU None, the i64::MIN case correctly returning i64::MIN), q32_mul negate-before-extract floor (60,014), the F7 negative-argument rounding pin, resident ping-pong equals one-shot, batched-worlds equals separate launches, and the cache read-path. Two open questions resolved: CubeCL lowers native i64 division bit-exactly on CUDA, and the ln coefficients work both inline and buffer-bound.

**Measured speedup (RTX 5090, CUDA, release, warm-up plus averaged).** Residency on a 256x256 field loop: ~38 ms to ~1.2 ms at N=64 (31 to 33x), ~137 ms to ~3.8 ms at N=256 (30 to 36x). exp fold 7.5 to 11.5x (1M elements). Batched worlds 11 to 13x (B=8, 64x64). ln fold ~1.1x, a near no-op, because NVRTC already constant-folds the ln coefficient divisions (both operands compile-time) while exp's division has a runtime dividend NVRTC cannot fold, so the real transcendental win is exp's native division, not ln's coefficient fold. Transfer-vs-compute for one 256x256 step: H2D ~0.45 ms, kernel ~0.015 ms, D2H ~0.05 ms, so transfers are 30 to 43x the compute, which is why residency is the dominant lever. These are micro-benchmarks; the real end-to-end tick speedup depends on the integrated pipeline and the per-tick workload mix.

**Implementation handoff, ordered.** (1) `div_checked` / `checked_div_zero` in `prim`, route `logistic_increment` through it, keep the `q2 != 0` and `at_min` terms, do NOT touch the transcendental/noise `q32_div` sites (do first, correctness, covered by T3). (2) fold `fixed_exp` to native division and `fixed_ln` to const coefficients, keep the 18/13 term counts and the floor shift (byte-neutral, T1/T2/T5). (3) the `WorldState` residency and per-tick launch graph (the structural win, ping-pong is a handle swap, no mid-tick readback, T6, plus a one-tick full-graph resident spot-check that is the single unrun validation item). (4) batched worlds via a z-slab offset (byte-neutral, T7). (5) the precompute-cache registry, with the CORDIC table and ln coefficients as inline comptime consts (universal constants) and the registry reserved strictly for per-run derived values from a profile (byte-neutral, T8).

**Five decisions surfaced for the owner.** (1) is the GPU the production compute path for the live tick loop, or a proven-parallel backend for batch work (this gates whether the ~30x residency win lands every tick or only on batch tasks); (2) the snapshot cadence, reserved, set by observation need not compute; (3) confirm the `q32_div` call-site audit boundary from change 1; (4) shared vs per-world per-axis params in batched mode; (5) keep the load-time cache verification assert on in release (Fable recommends yes, it is the memoization guarantee and is load-time cheap).

**The validation harness** lives in `crates/gpu/src/opt_test.rs` (test-only, gated behind `CIVSIM_GPU`/`CIVSIM_GPU_BENCH`, no production kernel modified); it is the regression suite for when a cloud engineer integrates the handoff.
