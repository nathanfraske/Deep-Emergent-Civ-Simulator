## Grounding notes (verified against the actual tree before synthesizing)

I read the real files rather than trusting the three designs' line numbers at face value. All three hardcode descriptions check out: `derive_region` (crates/sim/src/genesis.rs:705-752) matches exactly, including the two call sites at lines 447 and 556 both reading `params.generator.env_axes` (note the nesting: `GenesisParams.generator: GeneratorParams`, not a bare field, which matters for where T4's registry gets threaded). `AbioticSourceRegistry`/`AbioticBinding`/`AbioticField` in environ.rs match the T1 design's shape. `run_world.rs`'s hand-built registry sits at lines 1877-1884 as claimed, and a grep of `worldbuild.rs` confirms `build_dawn_runner` never calls `set_abiotic_sources` anywhere in the canonical path (it calls `set_producer_source` only), so the production gap T1 describes is real. `decompose.rs`'s `activity_at` is the fixed five-argument signature `(temperature, barrier, moisture, oxygen, life_stock)` at lines 285-305, and its one call site (runner.rs:3566-3589) reads `moisture_at`/`respirable_at` exactly as described, with the `unwrap_or(Fixed::ONE)` non-limiting convention intact.

One seam worth flagging before authoring anything: T4's design claims moisture "has no physics-floor identity at all." That is true of `Tile.moisture` specifically (it is pure fractal noise, `crates/world/src/noise.rs` `FIELD_MOISTURE = 1`, with no floor axis behind it, confirmed by grep), but `crates/physics/data/fluids_floor.toml` already carries a dormant `fluid.vapor_pressure` axis (id at line 136) whose `measures` field literally says "the moisture state the water cycle transports," with `laws::saturation_vapor_pressure` already implemented and already called once, from `environ.rs:595`, inside `EnvironFields::step`. It is not the same physical quantity as the volumetric water-content axis T4 proposes (vapor pressure in MPa versus a saturation fraction), so authoring a new `fluid.moisture_content` axis is still probably right, but the build agent must not skip checking whether the existing `fluid.vapor_pressure` axis (and the psychrometric law already sitting next to it) is the intended floor identity for "moisture" before adding a second one. I fold this into T4's build steps below as a required due-diligence step, not a blocker.

## Dependency order

Build T4 first, T1 second, T6 third (T6 can in practice run in parallel with either, since it is off the canonical path and touches no shared struct, but ordering it last keeps the plan linear and lets it consume T4's finished `ConditionSource`-style vocabulary as a template rather than duplicating the design work).

The reason T4 leads: it is the only one of the three that changes a shared data shape other consumers depend on. `Tile` (crates/world/src/worldgen.rs:32-37) currently has three named `Fixed` fields; T4 turns it into `axes: Vec<Fixed>`. That single change ripples into `TileMap::generate`, `TileMap::state_hash`, `BiomeSet::classify`'s call site, and, critically, `EnvironFields::from_map` (environ.rs:407-408), which reads `tile.elevation`/`tile.moisture` directly to build the fields `moisture_at` (and therefore T6's Conditions kernel) ultimately reads. If T1 or T6 is built against the old three-named-field `Tile` first, both would need a second pass once T4 lands. Building T4 first means T1's `AbioticAvailability` bands can be written directly against the final `EnvAxisRegistry` ordinal semantics (with a named axis, not a bare "axis 1 happens to mean moisture" convention), and T6's call site needs no rework since it only ever calls `moisture_at`/`respirable_at`, whose underlying data source T4 already stabilizes.

### Step 1 of 3: T4, env axes become a data-defined registry

**Current hardcode.** `crates/world/src/worldgen.rs` `Tile` (lines 32-37): three named `Fixed` fields (`elevation`, `moisture`, `temperature`) plus `biome`. `TileMap::generate` (~lines 86-116) makes exactly three fixed `fractal()` calls keyed to the literal salts `FIELD_ELEVATION=0`/`FIELD_MOISTURE=1`/`FIELD_TEMPERATURE=2` (crates/world/src/noise.rs:28-32), plus one `blend_latitude` call for temperature. `Tile::state_hash` hashes those three named fields in order (worldgen.rs:151-153). `crates/sim/src/genesis.rs` `derive_region` (lines 705-747) folds the same three named fields into block means, then hardcodes `fields = vec![m_elev, m_moist, m_temp, m_moist]` (line 734, the fourth "soil-fertility" slot is a bare duplicate of moisture) and pads any axis beyond four with the literal `Fixed::from_ratio(1, 2)` (lines 736-738). `crates/sim/src/biosphere.rs` `GeneratorParams::env_axes` (line 232, `dev_default` sets it to `4` at line 277) is a second, independently authored axis count that only agrees with the four-slot vector by convention. `EnvironFields::from_map` (environ.rs:407-408) is a third, independent direct reader of `tile.elevation`/`tile.moisture`.

**Data-defined replacement.** A new `EnvAxisRegistry` (new file, `crates/sim/src/envaxis.rs`, sibling to `AbioticSourceRegistry`), ordinal-indexed, one `EnvAxisBinding` per axis: `floor_axis: Option<String>` (the physics-floor `QuantityAxis` id this axis is physically identified with, or `None` for a pure-topology axis like elevation, exactly as a bare coordinate needs no floor entry), `source: EnvAxisSource` (a closed three-variant enum of value SHAPES, not domain names: `GeneratedField{field_salt: u32}`, `LatitudeBlend{field_salt: u32, weight: Fixed}`, `Uniform(Fixed)`), and `niche_range: Option<(Fixed, Fixed)>` (reserved, when `floor_axis` is `Some`, narrowing the floor axis's own wide authored range down to this world's biosphere span). `EnvAxisRegistry::len()` becomes the single source of truth for axis count, replacing `GeneratorParams::env_axes` outright rather than living beside it. `Tile` becomes `{ axes: Vec<Fixed>, biome: BiomeId }`; `TileMap::generate` loops over a world-declared axis-generation spec instead of three fixed calls; `Tile::state_hash` hashes the vector in registry order. `derive_region` folds `tile.axes` generically to a `registry.len()`-long mean vector, with its T1-owned abiotic block left untouched for now (T1 is step 2). `EnvironFields::from_map`'s hydrology consumers (`compute_downhill`, `is_basin`) get an explicit `elevation_axis: Option<usize>` role binding rather than assuming ordinal 0 is elevation. `BiomeSet::classify`'s call site reads whichever ordinals the world's spec designates for elevation/moisture/temperature, with the caveat (named, not solved) that a fully axis-agnostic biome classifier is out of scope; biome id is cosmetic and `Niche::suitability` never reads it, confirmed by reading biosphere.rs.

**New floor axis, and its legitimacy.** One new row, `fluid.moisture_content`, added to `crates/physics/data/fluids_floor.toml` (a fluids-domain state axis, sibling to the already-present `fluid.respirable_content`), with `range_lo`/`range_hi` surfaced as **reserved, not invented**, on the basis of Hillel's *Environmental Soil Physics* field-capacity-to-saturation volumetric water content bounds. This is a legitimate physics-floor addition under Principle 11 and CLAUDE.md Section 7: it closes a real coherence gap (today `Tile.moisture` is unmoored fractal noise with zero floor identity, confirmed by grep: no `moisture` axis exists anywhere in `ground_floor.toml`), and it is structurally identical to every other floor-axis row in the file. Before writing it, the build agent must first check and record, in the row's own comment, why the already-present dormant `fluid.vapor_pressure` axis (fluids_floor.toml line 136, "the moisture state the water cycle transports," with `laws::saturation_vapor_pressure` already implemented at environ.rs:595) is not the correct binding instead. My read is that they are legitimately different physical quantities (a saturation vapor pressure in MPa is not a volumetric water-content fraction), so a new axis is likely still correct, but this must be an explicit, cited decision in the commit, not a silent duplicate.

**Byte-neutral opt-in path.** A dev-default `EnvAxisRegistry` of exactly `[GeneratedField{salt:0}, GeneratedField{salt:1}, LatitudeBlend{salt:2, weight:3/5}, Uniform-or-moisture-duplicate]`, same salts, same order, same weight as today's hardcoded calls, reproduces `Tile.axes`, `derive_region`'s fold, and `TileMap::state_hash` bit-for-bit, provided the hash loop iterates in registry order. What legitimately re-pins (a signature change, not a value change): every call site reading `p.env_axes` (biosphere.rs `sample_candidate`, any test asserting `env_axes==4`), every direct `tile.elevation`/`.moisture`/`.temperature` read (`EnvironFields::from_map`, `derive_region`, `BiomeSet::classify`'s call site, all three confirmed by grep), and any test constructing `GeneratorParams` positionally.

**Build steps, in order.**

### Step 3 of 3: T6, the Conditions kernel's axis set becomes data

**Current hardcode.** `crates/sim/src/decompose.rs` `DecomposerKernelId::Conditions` (lines 98-103) has a fixed axis SHAPE baked into `DecomposerDriver::contribution`'s five-argument signature `(temperature, barrier, moisture, oxygen, life_stock)` (lines 164-196), unconditionally computing three saturating responses (moisture, oxygen, warmth-above-barrier) and folding by minimum (line 189). `DecomposerDriverRegistry::activity_at` (lines 285-305, confirmed by reading) and its one call site (runner.rs:3566-3589, confirmed by reading) carry the same fixed five-slot shape through, reading `moisture_at` (environ.rs, sourced from Tile via T4's now-fixed axis registry) and `respirable_at` (already general per the medium substrate). A world cannot drop, rename, or add a fourth axis without a new kernel variant or a signature change.

**Data-defined replacement.** A closed `ConditionSource` enum (sibling to `AbioticField`, same fixed-vocabulary-plus-data-binding boundary: `Moisture`, `Respirable`, `WarmthAboveBarrier`), a row-level `axes: Vec<ConditionAxis>` field on `DecomposerDriver` (`ConditionAxis{name: String, source: ConditionSource}`, empty for Life rows), and `contribution`'s Conditions arm rewritten as a plain Liebig-minimum fold over `self.axes` against a caller-supplied `profile: &[(ConditionSource, Fixed)]` slice, skipping any axis absent from the profile (non-limiting, the existing convention made strict rather than incidental). The warmth derivation moves out of the kernel into an exported `pub fn warmth_above_barrier(temperature, barrier) -> Fixed` helper (fixed Rust, a physics derivation, not data). `activity_at` drops its four named arguments for `(profile: &[(ConditionSource, Fixed)], life_stock: Fixed)`. The one call site in runner.rs builds `profile` by pushing `(Moisture, ...)` only if `self.environ` is armed, `(Respirable, ...)` only if the medium is armed, and always `(WarmthAboveBarrier, warmth_above_barrier(...))`.

**New floor axis or registry: none.** `ConditionSource` extends only when a genuinely new physical quantity enters the engine, the same bounded cost `AbioticField` already has (and which environ.rs's own module doc already names as deferred Arc-5 work). No new physics-floor row, no new top-level registry; `axes: Vec<ConditionAxis>` rides on the row structure `DecomposerDriverRegistry` already carries.

**Byte-neutral opt-in path.** Confirmed off the canonical run path today: `Runner::set_decomposer` has exactly one caller anywhere in the tree (`run_world.rs:1868`, under `--scenario full`), and `worldbuild.rs`'s `build_dawn_runner` never arms a decomposer registry. No world that omits `set_decomposer` is touched at all, by construction. `dev_fixture()` is rewritten to build its Conditions row via the new `ConditionAxis` list at the same three reference values (renamed keys, `moisture_saturation` to `moisture_reference`, `warmth_span` to `warmth_reference`, standardizing the naming), so its external behavior against the same raw inputs is unchanged, and `run_world.rs` needs no edit since it only calls the opaque `dev_fixture()`.

**Build steps, in order.**
1. Add `ConditionSource` (Moisture, Respirable, WarmthAboveBarrier) in decompose.rs.
2. Add `ConditionAxis` and `axes: Vec<ConditionAxis>` on `DecomposerDriver`; update the row constructors to build it, storing each axis's reserved reference under `"{name}_reference"` in the existing `params` map.
3. Extract `warmth_above_barrier` out of `contribution` as a public helper, keeping the saturating-subtraction bit trick unchanged.
4. Rewrite `contribution`'s Conditions arm to fold `self.axes` against a `profile` slice, dropping `temperature`/`barrier`/`moisture`/`oxygen` from the signature.
5. Update `activity_at` to `(profile: &[(ConditionSource, Fixed)], life_stock: Fixed) -> Fixed`; combine logic unchanged.
6. Update `dev_fixture()` to the new `ConditionAxis` list, renamed keys, same three reference values.
7. Update runner.rs's call site (lines 3566-3589) to build `profile` conditionally on `self.environ`/medium being armed, plus the unconditional warmth axis, then call `driver.activity_at(&profile, life_stock)`.
8. Rewrite decompose.rs's four existing unit tests to the profile-slice API, preserving intent; add one new test proving an axis absent from a row's `axes` list is non-limiting regardless of its raw value (the anaerobic-world proof).
9. Update the module doc and runner.rs's inline comments to describe the axis list as world data, not a fixed triad.
10. `cargo build -p civsim-sim`, `cargo test -p civsim-sim decompose::`, run `run_world --scenario full` to exercise the one production caller of `dev_fixture()`, and run the project's canonical determinism/consensus suite to confirm zero byte drift on any run that never calls `set_decomposer`.

## Adversarial check 1: derive-versus-author

Sweeping all three items for anything authored outside the physics floor:

- T4's `EnvAxisRegistry` rows (which axes exist, their generation shape, their salts and weights) are world data, not floor data, exactly like `GeneratorParams`/`GenesisParams`/`OrbitalElements` already are in this codebase. Legitimate: the floor governs physical-quantity identity, not which axes a given world's niche math walks.
- T4's one new floor row, `fluid.moisture_content`, is a legitimate floor addition (a genuine physical-quantity gap: `Tile.moisture` today has zero floor grounding, confirmed by grep of `ground_floor.toml`), but flagged above: the build agent must justify, in the commit, why it is not instead deriving from the already-present `fluid.vapor_pressure` axis and `laws::saturation_vapor_pressure`. This is the one real risk in the plan of authoring a value (or worse, a redundant floor axis) where reuse or derivation should have been checked first. It does not block the plan, but it is a required step, not optional due diligence.
- T4's `niche_range` per axis is correctly marked reserved with basis (a sub-band of the floor axis's own authored range), never invented in the build steps.
- T1 authors nothing beyond `AbioticAvailability` bands, which are world data (siting rules), not floor data; `available_in`'s fold is fixed Rust. No new number is invented anywhere in T1's build steps; the dev-fixture's `0.3` moisture threshold is the same value the current hardcode already used, carried forward unchanged, not a new invention.
- T6 authors nothing beyond the `axes` list and per-axis reference names, both world data; the Liebig fold and `warmth_above_barrier`'s arithmetic are fixed Rust reused from the existing pattern (`Niche::suitability`, `net_nutrition`). No reserved value is fabricated; the dev-fixture's three reference values are renamed keys carrying forward the same numbers, not new ones.

No step in the synthesized plan invents a number outside the floor. The one item requiring owner attention before it is fully clean is the `fluid.moisture_content` versus `fluid.vapor_pressure` reuse question in T4, which should be resolved (or at minimum explicitly documented) before that row is committed.

## Adversarial check 2: alien feasibility, walked through the built plan

Concrete world: a lightless, anaerobic, mana-fed subterranean biosphere.

**T4 (env axes).** This world's `EnvAxisRegistry` declares, say, three axes: axis 0, depth-below-surface (`floor_axis: None`, pure topology, `GeneratedField`, generalizing today's elevation the same way a cave world would); axis 1, ambient redox potential (`floor_axis: Some("chem.standard_potential")`, an existing chem_optics_floor.toml axis, `GeneratedField`), standing in for the light this world's producers never draw on, so chemosynthetic zonation emerges from niche fit exactly as photosynthetic zonation does on a lit world, through the same generic `Niche::suitability` fold (confirmed axis-agnostic by reading biosphere.rs); axis 2, mana flux (`floor_axis: Some("mana.flux")`, a new row this world's own `mana_floor.toml` adds, legitimate under Principle 9 since fantasy physics is an authored cultural input at the floor). No engine Rust changes: all three `EnvAxisSource` shapes are domain-blind, and `fractal()` already draws an unbounded number of independently salted streams. `Tile.axes` on this world is a three-long vector; nothing assumes four, nothing assumes ordinal 1 is moisture. Elevation-role binding (`elevation_axis`) can point at the depth axis if the world wants downhill routing, or `None` if it does not.

**T1 (abiotic presence).** This world's `GenesisParams.abiotic` simply omits any row bound to `AbioticField::Light`, so `available_in` never includes it in any region, with zero special-case code, since the mechanism only iterates whatever rows the world declares. It declares a redox-based producer source with a band on axis 1, and a mana-flux producer source with a band (or an always-available empty-bands row) on axis 2, each carrying its own floor nutrient class (say `chem.sulfide_reduced` and `mana.ambient`) rather than `bio.organic_residue`. Chemosynthetic and mana-fed producers ground exactly as photosynthetic ones do today, through the same fold, zero Rust changes.

**T6 (decomposition).** If this world wants matter breakdown gated by mana flux rather than moisture or oxygen, its Conditions row's `axes` list simply contains one entry, `ConditionAxis{name: "mana_flux", source: ConditionSource::ManaFlux}`, and drops `Moisture`/`Respirable`/`WarmthAboveBarrier` entirely. This is the one honest edge the plan does not fully close for free: `ConditionSource::ManaFlux` does not exist yet as a variant, so it needs one disclosed, one-time Rust addition (a new enum variant plus one new field-reader wired at the single runner.rs call site), the same bounded cost every genuinely new physical quantity has paid in this engine (light, water, soil, the medium substrate itself). If this world is content expressing decomposition purely through the anaerobic omission (no `Respirable` axis, keep `Moisture` and `WarmthAboveBarrier`, or even zero axes for fully unconditional decay), that is pure data, zero code change, and directly proves the anaerobic case the new unit test in step 3's build steps is written to cover.

No step in the walked-through plan silently assumes light, oxygen, water, or Earth-shaped terrain survives into this world's data. The only residual cost, `ConditionSource::ManaFlux` as a new enum variant, is named as a disclosed limit in the T6 design rather than hidden, and it is the same class of bounded extension `AbioticField` already carries as documented deferred work in environ.rs.

## Map (source-model)

SOURCE-MODEL MAP: biosphere.rs / genesis.rs / environ.rs / decompose.rs

1. The biosphere-layer model itself (crates/sim/src/biosphere.rs) is already axis-agnostic and id-opaque, and that is worth stating up front because it narrows where the real defect lives.

- `EnvProfile { fields: Vec<Fixed> }` (biosphere.rs:50-64) is an unbounded, unlabelled vector. `Niche::suitability` (biosphere.rs:95-128) walks `0..n` where `n = optimum.len().min(breadth.len()).min(env.fields.len())`, reading each axis purely by ordinal, with no field ever called "moisture" or "temperature" in code, only in comments. `Niche::distance` (biosphere.rs:134-145) is the same, axis-count-agnostic.
- `Region { env: EnvProfile, abiotic: BTreeSet<u16> }` (biosphere.rs:70-73) treats `abiotic` as an opaque set of source ids; nothing in biosphere.rs interprets what a given `u16` means.
- `SourceRef::Abiotic(u16) | SourceRef::Species(SpeciesId)` (biosphere.rs:81-86) is opaque by construction.
- `GeneratorParams::env_axes: usize` (biosphere.rs:232, dev default 4 at biosphere.rs:277) is already a data field, and `sample_candidate` (biosphere.rs:454-489) draws `optimum`/`breadth` for `0..p.env_axes` generically (lines 463-472) and, for a producer, picks one present abiotic id uniformly at random from `region.abiotic` with no id-specific branch (lines 474-481).
- `grounded()` (biosphere.rs:304-330), the least-fixed-point closure walk, only ever tests `abiotic.contains(a)` (line 317); it never inspects what `a` is.

So the generator, the niche-fit law, and the closure walk would already admit a world-declared axis count and a world-declared abiotic id set with no code change. The hardcoding is entirely upstream, in what actually constructs a `Region`.

2. Where the Earth abiotic triad is hardcoded (T1): crates/sim/src/genesis.rs, fn `derive_region`, lines 705-752, specifically lines 740-746:
```
let mut abiotic = std::collections::BTreeSet::new();
abiotic.insert(0u16); // light, always available above ground
if m_moist >= Fixed::from_ratio(3, 10) {
    abiotic.insert(1u16); // water, where the block is moist enough
} else {
    abiotic.insert(2u16); // a dryland soil-nutrient source, so producers can still ground
}
```
This is the sole production site that decides which abiotic ids exist in a region, and it hardcodes exactly the Earth triad (light always present; water xor a dryland soil id, gated on a literal moisture threshold `3/10`) with no per-world override parameter. `derive_region`'s signature (genesis.rs:705) takes only `(map, x0, y0, side, env_axes)`, nothing that could let a world supply its own abiotic scheme. Both call sites, `genesis()` (genesis.rs:447) and `WorldGenesis::new()` (genesis.rs:556), call this same function, so there is exactly one hardcode site for T1, reached identically from both the batch and staged driver.

A second, independent hardcode of the same triad exists at the far end of the pipeline, in crates/sim/examples/run_world.rs:1870-1884, where the `AbioticSourceRegistry` that the run consults is armed:
```
sources.insert(0, AbioticField::Light, false, "");
sources.insert(1, AbioticField::Water, true, "");
sources.insert(2, AbioticField::Soil, true, "bio.organic_residue");
```
Nothing ties this to `derive_region`'s choice of ids except comment discipline; the two sites agree only because a human kept them in sync ("0 light, 1 water, 2 soil" is asserted independently in both places). There is no shared data source of truth for "what does id N mean here." `crates/sim/src/environ.rs`'s own doc comment even says the binding is set "at the biosphere-arming site" (environ.rs:377), but that site is not derived from the biosphere generation at all; it is a second, hand-written literal registry.

3. Where the env axes are hardcoded (T4): also `derive_region`, lines 709-738. The map fold reads exactly three named `Tile` fields per cell (`t.elevation`, `t.moisture`, `t.temperature`, genesis.rs:715-717), means them over the block (lines 721-732), and then builds:
```
let mut fields = vec![m_elev, m_moist, m_temp, m_moist]; // elevation, moisture, temperature, soil-fertility(=moisture stand-in)
fields.truncate(env_axes);
while fields.len() < env_axes {
    fields.push(Fixed::from_ratio(1, 2));
}
```
Two distinct defects stack here. First, the content of axes 0-3 is fixed Earth terrain (elevation/moisture/temperature plus a moisture-duplicate stand-in for soil fertility, per the module doc at genesis.rs:29-30, "a moisture-derived stand-in until the soil `Stock` worldgen field lands"; I confirmed that field has not landed: `grep soil` over crates/world/src/*.rs returns nothing). Second, if a world sets `env_axes` above 4 (a value `GeneratorParams` already exposes as data), every axis beyond the fourth is padded with the literal constant `Fixed::from_ratio(1, 2)` for every region uniformly, so it carries zero differentiating information; a niche drawn against it is meaningless (same response everywhere). `env_axes` below 4 truncates the Earth vector rather than choosing a different, smaller set of world-appropriate axes.

This is itself downstream of a harder constraint: `crates/world/src/worldgen.rs`'s `Tile` struct (lines 32-37) is a closed Rust struct with exactly three named `Fixed` fields (`elevation`, `moisture`, `temperature`) plus a `biome: BiomeId`. `derive_region` cannot read anything else because `TileMap`/`Tile` do not carry anything else. So "a world declares its own floor axes" cannot be satisfied inside genesis.rs alone; the terrain-generation layer in `civsim_world` is itself Earth-terrain-shaped at the struct level, not just at the fold site.

4. How a producer's Abiotic id flows from generation to the run registry, traced precisely:

a. `derive_region` (genesis.rs:705-752) mints `Region.abiotic = {0} ∪ {1 or 2}` for every region.
b. `biosphere::generate` -> `sample_candidate` (biosphere.rs:361-435, 454-489) draws each layer-0 species's `draws_on = vec![SourceRef::Abiotic(id)]` by picking one id uniformly from `region.abiotic` (biosphere.rs:479-481). The id is carried opaquely from here on.
c. `LivingWorld::producer_sources()` (genesis.rs:169-200) walks placed occupants and, per producer, collects every `SourceRef::Abiotic(id)` edge into `Vec<(Coord3, Vec<u16>)>`, explicitly never interpreting the id's meaning (its own doc comment at genesis.rs:165-168 says so: "no `id == 2` soil switch").
d. `EnvironFields::set_producer_source()` (environ.rs:688-699) writes that `Vec<Vec<u16>>` into `producer_source: Vec<Vec<u16>>` per cell (still opaque ids). The one production caller of this pair is `worldbuild.rs:432`: `environ.set_producer_source(&living.producer_sources());`, inside `build_dawn_runner` (worldbuild.rs:300 onward), gated on `peoples.biosphere` being armed (worldbuild.rs:430-441).
e. `EnvironFields::extract_producers()` (environ.rs:747-833) is where the id finally acquires meaning: for each producer cell it looks up `registry.binding(id)` (environ.rs:776-778, panicking loud if a world seeded a producer source id the registry never bound), reads `binding.field` to select `AbioticField::Light | Soil | Water` (environ.rs:779-790) and `binding.depletes`/`binding.class` to decide whether and how to draw the field down (environ.rs:807-830).
f. The `AbioticSourceRegistry` that `extract_producers` consults is armed nowhere in the `build_dawn_runner`/`worldbuild.rs` production path; I grepped the whole tree for `set_abiotic_sources` and the only two callers are `crates/sim/src/runner.rs:3203` (the setter itself) and `crates/sim/examples/run_world.rs:1884`, called from the hardcoded triple insert at run_world.rs:1877-1880 quoted above. So today the registry that gives ids meaning is armed by a separate, hand-written example-only literal that happens to agree with `derive_region`'s literal by convention, not by any shared type or data file.

`AbioticField` itself (environ.rs:325-333) is also a closed three-variant Rust enum (`Light`, `Water`, `Soil`); it is not the u16-id layer (which is properly opaque data), but it is the fixed vocabulary of run *fields* an id can bind to. The module doc (environ.rs:318-323) is explicit that this is deliberately decoupled from depletion behaviour (`AbioticBinding::depletes`, `FINDING-1`) and gestures at "Arc 5's data-defined field set" as the point where an alien field (geothermal, redox, mana) becomes a new field handle rather than a new enum variant; that generalization has not happened yet, so a genuinely alien producer today is bounded by this three-member enum even though the id-to-binding mapping around it is already data.

5. The `decompose.rs` Conditions kernel (T6): `DecomposerKernelId::Conditions` (decompose.rs:98-103) and its `contribution` arm (decompose.rs:172-190) are fixed to exactly three named inputs, `moisture`, `oxygen`, and `warmth` (temperature above the substance's own decomposition barrier), combined by a Liebig minimum (decompose.rs:189). The function signature that carries them, `DecomposerDriver::contribution(temperature, barrier, moisture, oxygen, life_stock)` (decompose.rs:164-171) and `DecomposerDriverRegistry::activity_at` (decompose.rs:285-305), is fixed Rust; a world cannot add a fourth conditions axis (pressure, salinity, a redox potential, a mana flux) without a new Rust kernel variant. That is the real T6 hardcode: not a missing config value, but a closed kernel shape.

At the one call site, `runner.rs:3566-3587` (inside the matter-cycle beat), `moisture` is read from `self.environ.as_ref().map(|(env,_)| env.moisture_at(cell.x, cell.y)).unwrap_or(Fixed::ONE)` (runner.rs:3575-3579), and `EnvironFields.moisture` is populated in `EnvironFields::from_map` (environ.rs:395-420) by folding `tile.moisture` straight off the same Earth-terrain `Tile` struct discussed under T4 (environ.rs:408, `moisture.push(tile.moisture)`). So the moisture axis of the Conditions kernel is doubly Earth-bound: the kernel can only ever be handed a "moisture" number, and that number is sourced from the same closed `Tile.moisture` field `derive_region` reads.

By contrast, `oxygen` at the same call site is read from `emb.physiology.as_ref().map(|p| p.medium.respirable_at(...))` (runner.rs:3580-3584), which is `crates/sim/src/medium.rs`'s already-generalized ambient-medium substrate (medium.rs:16-28 states its own design intent: "aquatic, terrestrial, and exotic-medium life are one mechanism," reading only a data-defined `fluid.respirable_content`, never a medium label). So within the Conditions kernel's three fixed inputs, the "oxygen" slot is already Principle-9-compliant (an alien world supplies its own medium data and the same code path works), while the "moisture" slot is not (there is no data-defined hydrology substrate underneath it comparable to the medium substrate; it is the raw Earth `Tile.moisture` scalar). The comment at runner.rs:3568-3574 documents that an *absent* environ/medium reads as fully non-limiting (1.0), which is the correct "unmodelled axis is non-limiting" convention, but that convention does not help a world that wants a *different* limiting axis instead of moisture; it can only opt out of moisture, not substitute something else, without a new kernel variant.

6. What "a world declares its own floor axes" would need to touch, concretely, given the above:

- `crates/world/src/worldgen.rs`: `Tile` (lines 32-37) would need to carry a data-defined axis vector instead of three named `Fixed` fields, or `TileMap` would need a parallel per-cell axis-vector layer a world's worldgen params populate; this is upstream of everything else and is the actual root of T4, not `derive_region` itself.
- `crates/sim/src/genesis.rs`, `fn derive_region` (lines 705-752): once `Tile`/`TileMap` carry a data-defined axis vector, this function's body (the `elev`/`moist`/`temp` folds at lines 709-738 and the literal `{0, 1-or-2}` triad at lines 740-746) would be replaced by a fold over the world's declared axis set and a world-declared abiotic-id scheme (a `GenesisParams`-level or worldgen-level registry, sibling to `AbioticSourceRegistry`, read here rather than hardcoded). The `env_axes` padding-with-`1/2` branch (lines 736-738) would go away once the axis count is driven by real per-world data rather than truncating/padding a fixed 4-vector.
- `crates/sim/src/environ.rs`: `AbioticField` (lines 325-333) is the remaining closed enum in the otherwise-data-defined `AbioticSourceRegistry`; the module doc already names the fix ("Arc 5's data-defined field set", environ.rs:322-323) as a new field-handle-plus-environ-field-it-reads rather than a new variant. `extract_producers`'s match on `binding.field` (environ.rs:779-790, 821-829) would need to dispatch through that data-defined field set instead of a three-arm match.
- The registry-arming site: today the only place ids 0/1/2 are bound to `Light`/`Water`/`Soil` is the hand-written literal in `crates/sim/examples/run_world.rs:1877-1880`, disconnected from `derive_region`'s choice of the same ids. A real fix needs one shared source of truth: either `derive_region`'s replacement emits both the `Region.abiotic` set *and* the matching `AbioticSourceRegistry` rows together (so the ids are defined once, at world-build time, and the run consults exactly what generation produced), or both read a common world-declared abiotic-scheme data structure. `crates/sim/src/worldbuild.rs`'s `build_dawn_runner` (around line 429-441) would be the natural place to arm that registry from the biosphere, since it already wires `producer_biomass`/`producer_sources` from `peoples.biosphere` but never calls `set_abiotic_sources` at all in production; that omission means the extract-deplete cycle only actually threads through end to end in the `run_world.rs` example path (which happens to hardcode the same triad convention), not in `worldbuild.rs`'s canonical build path.
- `crates/sim/src/decompose.rs`: `DecomposerKernelId::Conditions`'s fixed three-input shape (lines 98-190) and the `moisture_at` read at the call site (`runner.rs:3575-3579`, sourced from the same closed `Tile.moisture`) would need either a data-defined hydrology/axis substrate parallel to the medium substrate (so "moisture" becomes one instance of a world-declared limiting axis rather than a hardcoded field name), or a genuinely axis-agnostic Conditions kernel that takes a data-defined list of (axis-value, saturation-reference) pairs and Liebig-minimums over whatever a world declares, the same way `Niche::suitability` in biosphere.rs already does over `EnvProfile.fields`. The latter would also make the Conditions kernel reuse the same axis vector `derive_region`'s replacement produces, closing the loop between "what a region's environment is" and "what gates its decomposition."
